Proposal for Development of a Standardized Admin UI for Haneke Web Projects

Problem

The majority of Haneke's development projects either include an admin panel from the outset, or eventually evolve a necessity for one eventually. In fact, some of our projects — such as the Cura Transportation Management System — are essentially only an admin panel. This presents a problem where many of our clients would greatly benefit from the production of a custom admin panel, but paying for ground-up development of one is cost-prohibitive.

Current Solutions

To address this, we've used a variety of tools, services, and libraries to alleviate this including Strapi, WordPress, and various Ruby on Rails admin-UI gems. All of these generally have the same fatal flaws: customizability and extensibility. Many of them will allow for adjustments to the core functionality only through convoluted and restrictive plugin-based systems that make the development process for extending the core functions restrictive, painful, and/or impossible.

Proposed Solution

The proposed solution aims to address the need for a standardized Admin UI for Haneke web projects. By creating a custom Admin UI built in React, we can provide clients with a highly customizable and extensible template for their admin and dashboard user interfaces. This solution will alleviate the cost-prohibitive nature of ground-up development for each project and offer a more efficient and streamlined approach to building admin panels.

Overview

Initial Setup

  1. A Storybook component library would be established to create the individual components of the system.
  2. An Adobe XD and Zeplin Template stylesheet and design will be created based on the component library.
  3. A standard React template project would be created that would use the Storybook component library.

Implementation

  1. When the Product Strategy team is determining functionality for new projects, they can use the template project to determine what functionality exists within the current admin template, and what does not fit the template design.
  2. The design team uses the Adobe XD/Zeplin template in order to create a fully custom design for the client.
  3. The development team will clone the React template project, using the new Zeplin design and stylesheet, modify the storybook components to match the customized designs.
  4. Any new functionality can be abstracted into a generalized and reusable component for future projects and saved to the React template.

Maintenance

  1. Each copy of the templated project becomes a version of it frozen in time and maintained independently with the client's maintenance budget.
  2. The main template repository is independently updated with the most useful parts if and when they are deemed beneficial for future projects.

Future-proofing

  1. The use of React as the basis for the platform ensures that we're building upon a well-supported and reliable framework that will likely be around for many years to come.
  2. Building the frontend UI to be backend-agnostic allows the backend to be swapped out at will with any backend as needed, or for the application to be made full-stack.
  3. Having the component library be housed within a system such as storybook, and for the designs to be tightly coupled to that library, allows for React to be swapped out for any frontend framework.

Initial Setup

Storybook

What is StoryBook?

Quote

Storybook is a frontend workshop for building UI components and pages in isolation. Thousands of teams use it for UI development, testing, and documentation. It’s open source and free.

More Info: Why Storybook?

Headless UI Library

To avoid unneeded development overhead building standardized components from scratch, the most logical choice is to use a UI library. The downside to most traditional UI libraries is that they are very restrictive on their styling and customization. The solution I propose here is to use something like React Aria from Adobe, Radix UI Primitives, or any other "Headless UI Library".

These libraries provide accessible UI primitives for your design system and offer behavior, accessibility, and interactions without imposing any rendering, DOM structure, styling methodology, or design-specific details. This allows us to focus on design and customize the components as needed. React Aria specifically ensures consistent behavior across different UIs, supports various interactions (mouse, touch, keyboard, screen reader), and has been tested across different browsers, devices, and platforms.

Customizability

To allow for this UI to be easily customized, I propose we use Storybook to build out the library of individual components that make it up. This will allow us to build a very basic template that can be changed extensively by our design team later to match the client's aesthetic and branding. We can build these components out of the Headless UI Library primitives, and add some default styling that can be entirely reconfigured based on the needs of the client.

Documentation & Testing

Another advantage of this is documentation. Storybook will allow us to fully document and test the individual components and parts of the system to both ensure future developers are able to quickly get up to speed and hop in and start building, and for us to ensure we've build a robust UI that we can rely on to work consistently.

Source: Why Storybook?

✅ Test UIs with less effort and no flakes

Stories are a pragmatic, reproducible way of tracking UI states. Use them to spot-test the UI during development. Storybook offers built-in workflows for automated Accessibility, Interaction, and Visual testing. Or use stories as test cases by importing them into other JavaScript testing tools.

Reusability

Building the interface from the ground up out of small well-tested components will allow us to create a system for building out more custom functionality later for extending the admin panel to fit the specific use-cases of our customers. If one customer requires a map screen in the admin panel, we can build the majority of the UI out of the pre-defined components, leaving the development of the map UI as the only piece of significant development work.

Taking this one step further, if we continue the concept of breaking up the UI into components, the aforementioned map can be built first in Storybook before being implemented for the client, and then added to our repertoire of components for reuse in future projects, creating a living document that continually grows in robustness as we use it.

Atomic Design

The concept of atomic design is not foreign to Haneke, as we've previously used it to create a design system for Frank Crum. Our recent meeting with the folks at Southleft highlighted how powerful this concept can be when paired with a system like Storybook. If we are smart about how we build our Storybook components, we can have a flexible and robust system that allows us to reduce development overhead and cost.

Starting with the smallest building blocks and working our way up into larger components, templates, and pages, will allow us to create a design system that can be tweaked down to the "Atomic" level, while maintaining design consistency across the project. We will accomplish this with a collaborative effort between design and development.

What is Atomic Design?


Source: Atomic Design | Brad Frost

Atomic Design is a methodology for creating design systems that focuses on breaking down user interfaces into smaller, reusable components. It consists of five distinct levels:

  1. Atoms: The basic building blocks, such as HTML tags (buttons, forms, inputs) and abstract elements like colors and fonts.
  2. Molecules: Groups of atoms combined to create more complex and functional units, like form inputs and labels together.
  3. Organisms: Assemblies of molecules working together to form recognizable sections of a user interface, like a navigation bar.
  4. Templates: Groups of organisms arranged in a specific layout to create pages or sections of a website.
  5. Pages: Specific instances of templates, showcasing the final design with real content.

Advantages of the Atomic Design methodology include:

  • Modularity: Design systems built using atomic design are modular, making it easier to manage and update different parts of a website independently.
  • Consistency: By using consistent atoms, molecules, and organisms, design systems ensure a cohesive look and feel across the entire interface.
  • Reusability: Components developed using atomic design can be reused throughout a website or across different projects, saving time and effort.
  • Scalability: Atomic design allows for easy scaling as projects grow, by adding new components or modifying existing ones.
  • Collaboration: It provides a common language for designers, developers, and stakeholders, enhancing collaboration and communication.
  • Efficiency: Atomic design streamlines the design and development process by breaking it down into manageable chunks, reducing redundancy.
Collaborative Approach to Initial Development

Here I suggest we take a collaborative approach, working together between the design and development teams to decide on the functionality that will influence design. This will allow for the design and dev teams to agree on an initial base design and functionality that will create a framework for the designers to build new versions of the UI.

Through this collaborative approach the initial component library is formed. With regular input from both teams, we come up with a storybook that describes the functionality of the admin panel. The basic functionality can be shown in the storybook and the components will be tested and robust with basic styling. Once this is completed, this storybook will form the basis for creating a template AdobeXD.

Adobe XD and Zeplin Template File

Storybook Informs Design Template

Once the initial Storybook is complete, we will likely have a Adobe XD/Zeplin file that roughly approximates the components within the Storybook. Here, we take this file and organize and standardize it to reflect the final project, so that it can easily be used as a template from which to customize the interface to a client's needs. This template should be a 1 to 1 of the storybook, tailored specifically to the needs of the design team.

Flexible, Extensible

Once the template has been created, matching the functionality of the storybook, it can be used as the basis for customizing the design. Furthermore, if the design team decides to modify this template for the design, it is immediately obvious if this is new or different functionality and can be flagged as such early to avoid surprises. Since these components are ultimately just plain React components and are thoroughly tested, extending or modifying behavior is trivial — especially since it can be flagged early and will not come as a surprise.

React Template Project

The Full Project Template

After completing the Storybook and XD Template, we can proceed to create a comprehensive template project. This project will include all the necessary frontend components and functionality, providing a complete solution for the project.

The React Advantage
Onboarding Speed

With React being the most popular frontend framework, using it as the template for the UI grants us the advantage of having fast onboarding. While there may be some level of learning curve due to Storybook, ultimately the components are simple React components, meaning that it should be easy to hop in and get work done with minimal onboarding time for new contractors.

Source: State of JavaScript 2022: Front-end Frameworks


Simplicity

This also allows for the framework itself to be malleable — modifying the framework is no different from development on a regular React application. This, coupled with the component library having testing built-in, should make any new changes easy to implement, easy to write tests for, and easy to verify that they don't break existing functionality.

Ecosystem

Using the most popular frontend framework also mean having access to some of the best-supported and tried-and-true libraries for implementing large chunks of functionality. State Management, Asynchronous Data Fetching, Application Routing, Tables, Forms, and more can be handled by long-standing open-source libraries that are well-documented and supported by a dedicated team and community.

Backend Agnostic

The idea here is that the frontend is implemented in a way that is Backend Agnostic. We can swap out backends as needed to suit the needs of the client. Leveraging tools like Tanstack Query React and/or GraphQL, we can build a system that simply expects certain data to be supplied to a particular function and for that function to return an expected payload, without being concerned how the data is being supplied.

Example

Tanstack Query React uses a React Hook called "useQuery" simply takes as an argument that either resolves with the data or throws an error. Leveraging this as the framework for the functionality of the application means that we simply need to ensure that we're supplying the right data to those functions — the implementation of how the data is sent or received is irrelevant.

Implementation

Definition

Business as Usual

When we're in the definition stage of a project, this template serves as one of the many tools in our tool belt. Product Strategy is able to assess whether or not the template makes sense for any given client, and whether or not it works out of the box, will need custom work or extending, or if it's not a fit for the project.

Early Flagging of Custom Modification

As the Product Strategists are determining the fit of the template to a clients needs, they can look at the full breadth of functionality available by using the template react project to see how it all fits together. If they need to dig in more, they can do so by perusing the Storybook to inspect the individual components to determine how well the clients needs align with what is already there.

From there they can easily determine the fit of what the client needs and whether:

  1. The basic template can simply be given a fresh coat of paint and shipped as is.
  2. The template contains most of the functionality, but will require modification of various components to make it work.
  3. The template contains some of the functionality, but whole new functionality will need to be added on top to accomplish what the client needs.
  4. The template does not match the client's requirements at all, and a ground-up development effort makes the most sense.

This takes a lot of the guess work out of their job by allowing them intimate access to the system to make their decisions with little to no barrier to entry. It also allows them to easily determine where the template and the final product will differ and document that early, reducing surprises later on.

Hosting a Generic Template

I propose that the finalized template project is hosted in a way that is accessible to the Product Strategy team. This would be a full-featured, functioning version of the project, and the accompanying StoryBook. This makes determining existing and new functionality at the outset easy for the PS team.

Design

Business as (Mostly) Usual

The design team will take the User Stories and notes from the Product Strategy team and complete a design sprint, creating a beautiful UI for the client as usually. The only change is that this design will be created based on the template that was created previously and will adhere to the functionality laid out in the original Storybook.

Collaborative Additions

If the product strategy team has flagged in the previous sprint modifications that will be needed, these changes will be collaborated on between the dev and development teams, ultimately culminating in:

  1. A decision on whether the additional functionality can and should be generalized and added to the Storybook and React Template, or if it should be a one-off implementation for only the current client.
  2. How this is implemented, what the final storybook component for this should look like, and how it should function.

If it is decided that this should be generalized and fed back into the system for future projects, the same process should be taken as the initial development of the template, but just isolated to this new component.

Development

Not Starting from Zero

The development team will start their part of the process by cloning the template project, meaning that, ideally, a large part of the work has already been completed. From here, they will simply need to modify the styling of the storybook components to match the final design from Zeplin. Once the styling is set up, the dashboard would just need to be hooked up to the backend and tested by QA.

Modifications and New Functionality

Since the components are pre-built and have testing already built in, modifying existing functionality is simplified. Developers can easily verify that the modifications don't break existing functionality and can write test that encompass only the new functionality. Additionally, the styling and modifications should automatically be reflected in the UI, with only minor adjustments to styling being needed to make it all function seamlessly.

Similarly, adding new functionality or components is also made easier. Developers can focus on those parts and trust that if their changes inadvertently break things, the testing structure set up will likely catch it.

Case-by-Case Template Additions

The team can decide on a case-by-case whether or not any project-specific functionality should be added to the parent template project. In the case that it doesn't make the cut, these additions can be made in the project without concern for generalizing them to work for other use-cases late. If it is decided that they make a good candidate for being added to the template, we can create them as a component in storybook and port that component back to the parent template project's storybook.

Example

For instance, let's imagine a client needs a map component on one of the pages of their dashboard. The team would come together and decide if that's something worth abstracting into a reusable component for later. Since a map is likely to be something that is needed, it gets added to the template project and a generic version first, then is imported and customized for the specific client.

The client gets a robust UI component, and the template grows to include another feature that doesn't require custom development.

Maintenance

Separation of Concerns

Freeze Frame(work)

Once the project is completed, it becomes a standalone instance frozen in time — meaning that it can be independently maintained. This allows each project have its own maintenance budget and agreement with the client for maintaining it, just like every other project. It's just another project — only it was built much faster.

Reduction of Overhead, Responsibilities, and Liability

Having each project use its own dedicated maintenance budget and be independently kept up to date reduces the burden of maintaining and versioning the Storybook UI and template. Once the project has been completed, it's its own self contained project. When the main template updates, we don't have to worry about conflicts with every project we have used it in.

Ever-Improving System

Growing Template

When a client needs specific new functionality, the team can come together and make a decision if it's general enough to be a good candidate for the template project. When we decide that it is, we go through the Initial Setup phase again, but just for that component, generalizing it so that it's reasonably flexible and can be reused. This allows the template to grow and become more full-featured and robust as time goes on.

Onward and Upwards

The template project is free to improve, grow, and expand independently of all of its children. New features can be added whenever without fear of breaking any existing projects. Every time a new project is started, it used the newest version of the template, meaning it is even more full featured out the gate, and has the potential to even make the next project even faster to spin up.

Future-proofing

A Strong Foundation

React isn't going anywhere any time soon. It's widely used and flexible with a massive community and tons of support. It's so popular that other frame works emulate its syntax and add in compatibility options to make it interop with them. Using React as the basis for the framework means that the foundations are solid, and are future proofed to an extent right out of the gate.

Source: Framework reimagined for the edge! 📚 Qwik Documentation

Flexible By Nature

While the easiest way for us to spin up this project might be a Ruby on Rails backend like most of our projects, the backend-agnostic structure allows for the backend to be swapped out for any backend. It can be Ruby, Firebase, Supabase, Node.js, or .Net. The frontend doesn't care.

To take this one step further: the framework, with a bit of additional configuring, could be moved over to a full-stack framework like Next.js, Remix, or Qwik. All that would need to take place is a reconfiguring of the components as outlined in the next section, and implementing the server-side code alongside the frontend code. This is a bit outside of the scope of this report, so I will leave it at that. The important thing is that this is a possibility.

Just The Beginning

Since the structure leverages Atomic Design and the Storybook informs the Design Template, the react version is just one version. While it would obviously require more work, the vast majority of the logic would be already in place for the system to be converted to any frontend framework. The development team can systematically go through the components in the Storybook and convert them to a different framework like Svelte, Solid, or Vue. The system being broken into manageable chunks simplifies this process.

Furthermore, the use of the Atomic design lends itself well to the potential of expanding this idea into micro-frontends, with each component being its own self-contained micro-app that can be imported into any project. This is a new concept that has been very slowly gaining traction in the frontend world; should it truly take off, the modular nature of the system is ready.

Conclusion

This proposed solution of creating a standardized Admin UI using React and Storybook for Haneke web projects offers a highly customizable and extensible template for admin and dashboard user interfaces. By leveraging a headless UI library and atomic design principles, the development process is streamlined, allowing for faster onboarding, simplified modifications, and the potential for future expansion into different frontend frameworks. The separation of concerns and maintenance of individual projects ensures efficient upkeep, while the continuous improvement of the main template repository ensures a strong foundation and adaptability for future projects. Overall, this solution provides a flexible and future-proof approach to building admin panels that meet the unique needs of Haneke clients.
